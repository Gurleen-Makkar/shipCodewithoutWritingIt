The Foundation: LLM Models

The "Pre-Agent" Era: Pre-Defined Control Logic

User wants a travel plan for Goa

User enters Goa as destination and selects the dates

System gathers all data (flights, hotels, etc.)

LLM receives all data to generate the plan

Here, the path is rigid. The program gathers all information upfront and feeds it to the LLM in one go.

The Tool Integration Problem

Before MCPs: The N×M Complexity

The Old Way

Every LLM provider had their own function calling way:

GPT: Custom JSON schema for Jira
Claude: Different JSON for same Jira tool
Gemini: Yet another format for Jira

Result: 3 tools × 3 providers = 9 implementations!

The MCP Way

One standard protocol for all LLMs:

Single Jira MCP Server
Standard MCP Protocol
All LLMs connect seamlessly

Migrate from Jira to Trello? Just swap the MCP server!

Think REST APIs for AI Tools

Just like REST standardized web APIs (GET, POST, PUT, DELETE),
MCPs standardize how LLMs interact with tools

Where Do Humans Fit In?

What to Automate

  • Initial code generation & boilerplate
  • Branching, commits, and PR creation
  • Ticket status updates and comments
  • Initial analysis and data gathering (RCA)
  • Repetitive, predictable tasks

What to Control

  • Final code review and approval
  • Complex architectural decisions
  • Confirming high-level designs (HLD/LLD)
  • Defining initial requirements (the Jira ticket)
  • Handling novel, unexpected problems

The goal is not to replace developers, but to supercharge them.

Key Takeaways

1. Developer workflows are shifting from assistance to automation.

4. The human role shifts to oversight, review, and high-level strategy.
5. AI-first engineering isn't theory; it's a practical way to drastically cut delivery times.
